# 项目记忆系统（Project Memory）

## 概述

WorkBuddy 没有原生跨会话记忆机制。本 Skill 在沙箱内自建 MEMORY 系统，通过 `MEMORY.md` 文件 + Git 持久化，实现 AI 的长期记忆。

**核心原理**：
- **读**：每次会话第一步，强制读取 `/workspace/MEMORY.md`
- **写**：每次回复前检查是否需要更新 MEMORY.md，如有则 commit + push
- **查**：关键决策点（改代码前、push 前）检查 MEMORY 中的规则约束

---

## MEMORY.md 标准结构

每个使用本 Skill 的项目，在 `/workspace/MEMORY.md` 中维护以下 6 个固定节：

```
## 1. 项目身份
- 项目名、产品定位、核心用户、IP 形象、Git 仓库等

## 2. 当前流程位置
- 产研流程进度条（PRD → 设计 → 原型 → UI → 技术方案 → 代码 → 测试 → 上线）
- 当前卡在哪一步
- 已交付文件清单

## 3. 用户偏好与决策
- 流程纪律（如：严格按产研流程走，不许跳环）
- 沟通偏好（如：每次询问 ≤3 个问题，业务决策先确认再执行）
- 设计偏好（如：状态栏极简，补卡不限 7 天）
- 技术决策（如：连续天数按自然日 00:00 重置）

## 4. 踩过的坑（规则变更日志）
- 每条含：日期、问题、根因、规则、对应设计清单编号
- 格式：表格（# | 日期 | 问题 | 根因 | 规则 | 对应清单）

## 5. 关键规则速查
- 与 design-spec §8.1 同步的交互逻辑清单
- 每次改代码前对照检查

## 6. 下一步
- 待用户确认的后续行动
```

---

## 三条铁律

### 铁律 1：读（每次会话第一步）

```
新会话启动 → 第一步 Read /workspace/MEMORY.md → 确认身份/流程/偏好/规则 → 再回复用户
```

**绝不跳过**。如果 MEMORY.md 不存在，告知用户并建议初始化。

### 铁律 2：写（每次回复前检查）

回复用户之前，遍历写入检查清单：

```
☐ 流程位置变了吗？（阶段推进/环节切换）→ 更新 §2
☐ 用户有新偏好或决策吗？→ 更新 §3
☐ 踩了新坑吗？（bug / 教训 / 意外行为）→ 更新 §4
☐ 有新规则需要加入速查吗？→ 更新 §5
☐ 下一步变了吗？→ 更新 §6
```

**只要有一个 ☐ 是"是"，就必须更新 MEMORY.md + git commit + git push。**

### 铁律 3：查（关键决策前对照）

在执行以下操作前，先查 MEMORY §3/§4/§5：

- **改代码前**：检查 §3.2 沟通偏好（是否需要先给方案让用户确认？）
- **push 前**：检查 §5 交互逻辑清单（是否会产生回归？）
- **做业务决策时**：检查 §3.3/§3.4（用户已确认过的偏好是什么？）

---

## 踩坑日志格式

§4 的每条记录必须包含以下字段：

| # | 日期 | 问题 | 根因 | 规则 | 对应清单 |
|---|------|------|------|------|----------|

**示例**：
```
| 14 | 2026-07-11 | calcStars 打卡 +2 星 | 今日 done 在 checkinsByDate 和 homeTasks 各算一遍 | 今日从 homeTasks 实时计算（去重），历史日累加 | — |
```

---

## 关键规则速查格式

§5 的每条规则使用简短标签（如 L1, L1.1, L8.4），与 design-spec §8.1 同步镜像：

```
- L1：今天不显示「补」标签
- L1.1：补卡模式打勾自动标 backfill
- L4：未完成任务直接打卡（无需二次确认）
```

---

## 初始化 MEMORY.md

首次使用时，运行 `scripts/init_memory.py` 或手动创建 MEMORY.md 模板：

```bash
python3 scripts/init_memory.py
```

模板会包含完整的 6 节骨架，AI 和用户共同填充初始内容。
