# 代码修改流程管控（Code Change Workflow）

## 概述

在 AI 辅助编程中，最危险的行为就是 AI 跳过用户确认直接修改生产代码。本 Skill 强制执行一套不可跳过的两步确认流程。

**核心问题**：AI 在修改代码时容易"顺手"把改动直接推到线上，用户来不及 review。

**解决方案**：所有代码修改必须经过两个确认点，缺一不可。

---

## 两步确认流程（铁律）

```
用户提需求
    │
    ▼
① 方案确认 ──── AI 先出方案，说明改什么、为什么这样改
    │
    ▼ (用户点头)
② 本地修改 ──── 修改非生产文件，或先备份；展示 git diff
    │
    ▼ (用户确认 diff)
③ commit + push ──── 用户确认后才上线
```

---

## 三条铁律

### 铁律 1：业务决策必须先确认

**凡是涉及业务逻辑、交互行为、文案、视觉变化的修改，必须先给方案，用户确认后才动手。**

❌ 错误：用户说"把火焰图标逻辑改一下" → AI 直接改代码
✅ 正确：AI 先解释"火焰图标当前逻辑是 X，建议改为 Y，效果是 Z" → 用户确认 → 改代码

### 铁律 2：代码修改必须展示 diff

**每次修改后，必须在 commit 前展示 `git diff` 给用户核对。**

❌ 错误：改完代码直接 `git commit -m "fix" && git push`
✅ 正确：改完 → `git diff` → 展示变更 → 用户确认 → commit + push

### 铁律 3：单文件项目尤其要小心

**对于"一个 index.html 就是整个产品"的项目（如 GitHub Pages 单页应用），修改即上线，必须格外谨慎。**

额外规则：
- 修改前先在本地做备份或使用临时文件验证
- 确认用户看到效果后再覆盖 index.html
- 绝不跳过第二步确认

---

## 检查清单（每次改代码前自问）

```
☐ 这是业务决策吗？→ 先出方案，等用户确认
☐ 改的是生产文件（index.html）吗？→ 先改副本，确认后再覆盖
☐ 改完了吗？→ 展示 git diff，等用户确认
☐ 用户确认了吗？→ 才能 commit + push
```

---

## 常见违规场景与纠正

| 场景 | 违规行为 | 正确做法 |
|------|---------|---------|
| 用户说"把最长连续天数去掉" | 直接删代码 + commit | 先确认：是去掉显示还是去掉整个逻辑？确认后再改 |
| 发现 bug 直接修 | 修完就 push | 先说明根因和修复方案，用户确认后再 push |
| 加个小功能 | 顺手加了就上线 | 功能再小也是业务决策，先方案再执行 |
